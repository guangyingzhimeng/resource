# Anthropic重磅发布Managed Agents：重新定义AI代理架构的未来

> 一大早又学习新东西了。anthropic 发了一篇新 blog 来讲他们最新发布的 Managed Agent 设计。

引发了社区的热议。短短数小时就获得了近百个赞。为什么这篇文章让技术圈如此兴奋？让我们深入探讨Anthropic的Managed Agents设计哲学。

## 问题：当模型的假设成为枷锁

传统的Agent Harness有一个根本问题：它把许多"模型做不到"的假设写死在代码里。

就像Anthropic在文章中提到的例子：Claude Sonnet 4.5会提前完成任务，因为它感知到上下文窗口正在接近——这种行为被称为"上下文焦虑"。工程师们通过在harness中添加上下文重置来解决这个问题。

但当同样的harness用在Claude Opus 4.5上时，这种行为消失了。那些重置逻辑变成了无用的死重。

这是一个普遍现象：**模型能力在飞速提升，但基于模型"做不到"的假设写死的系统，反而变得越来越笨重。**

## 灵感来自计算机科学的经典智慧

面对这个挑战，Anthropic团队回到了计算机科学的一个经典问题：**如何为"尚未想到的程序"设计系统？**

答案来自几十年前操作系统的设计经验。通过将硬件虚拟化成抽象概念（进程、文件），操作系统的设计持续了数十年。read()命令并不关心它访问的是1970年代的磁盘组还是现代的SSD。上层的抽象保持稳定，底层的实现可以自由改变。

Managed Agents遵循了同样的模式。

## 三大组件：架构的艺术

Anthropic将Agent虚拟化为三个核心组件：

### 1. Session（会话）
一个追加式日志，记录发生的一切。它不是Claude的上下文窗口，而是外部存储的持久化事件流。

### 2. Harness（驱动框架）
调用Claude并将Claude的工具调用路由到相关基础设施的循环。

### 3. Sandbox（沙箱/执行环境）
Claude可以运行代码和编辑文件的执行环境。

**最重要的是，这三个组件的实现可以相互替换而不影响其他组件。**

```
Brain（Claude + Harness）
    ↓
Hands（Sandbox + Tools）
    ↓
Session（事件日志）
```

## 从"宠物"到"牛群"的进化

最初，Anthropic将所有组件放在一个容器中。这带来了便利，但也产生了一个基础设施的经典问题：他们养了一只"宠物"。

在基础设施的"宠物vs牛群"比喻中，宠物是具名的、需要精心照料的个体，你不能承受失去它；而牛群是可以互换的。

容器成了那只"宠物"。如果容器失败，会话就丢失了。如果容器无响应，就必须"治愈"它回到健康状态。更糟糕的是，由于容器还持有用户数据，工程师甚至无法有效地调试。

## 关键突破：大脑与手的解耦

解决方案是**将"大脑"（Claude和Harness）与"手"（Sandbox和工具）以及"会话"相互解耦**。

每个组件都成为一个接口，对其他组件做出最少的假设，每个都可以独立失败或替换。

这种解耦带来了巨大收益：

### 1. 容器变成了"牛群"
当容器死亡时，Harness捕获工具调用错误并传递给Claude。Claude决定重试时，新容器可以用标准配方重新初始化。不再需要"治愈"失败的容器。

### 2. 故障恢复变得简单
由于会话日志位于Harness之外，Harness中的任何内容都不需要在崩溃后存活。当一个失败时，新的可以通过wake(sessionId)重新启动，使用getSession(id)取回事件日志，并从最后一个事件恢复。

### 3. 安全边界得到加强
在耦合设计中，Claude生成的任何不受信任代码都在与凭据相同的容器中运行。解耦后，确保令牌永远不会从Claude生成代码运行的沙箱中访问到。

## 性能的飞跃

这种架构改变带来了显著的性能提升，特别是在**time-to-first-token（TTFT）**上——这是衡量会话接受工作到产生第一个响应令牌之间等待时间的指标，也是用户最敏锐感受到的延迟。

- **p50 TTFT降低约60%**
- **p95降低超过90%**

这是因为在解耦架构中，容器只有在需要时才通过工具调用（execute(name, input) → string）由大脑配置。不需要容器的会话不会等待容器设置。

## 多大脑，多手的未来

解耦还打开了新的可能性：

**多大脑**：当团队希望Claude在他们自己的VPC中处理资源时，只需要将Harness连接到他们的环境，而不是网络对等。扩展到多个大脑只需要启动多个无状态的Harness，只在需要时连接到手。

**多手**：每个大脑可以连接到多个手。每个手都变成一个简单的工具接口：名称和输入进入，字符串返回。Harness不知道沙箱是容器、手机，还是宝可梦模拟器。

## 未来的启示

Managed Agents的设计哲学给了我们重要启示：

1. **避免过度假设**：不要基于模型当前的局限性写死系统设计
2. **拥抱解耦**：让系统各部分独立演化，而不是紧密耦合
3. **思考持久性**：什么抽象能够经受住时间的考验？
4. **性能即体验**：架构优化最终要服务于用户体验

正如kevinma_dev_zh在结尾的洞见："系统里最容易过时的往往不是模型本身，而是围绕模型写的那层Harness。"

在一个AI能力飞速发展的时代，也许最重要的设计原则是：**构建一个为未来而生的系统，而不是为现在而生的系统。**
